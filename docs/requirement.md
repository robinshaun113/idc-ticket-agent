# IDC 智能工单分派与诊断 Agent · 需求文档

## 用户画像

**IDC 一线运维工程师**（初级为主），夜班值班时常面临"不知道该找谁、不知道该查什么、不知道该不该叫醒主管"三个决策困境。

**IDC 运维主管**，关注工单响应时长、误派率、重复工单率，希望系统能沉淀隐性经验而不是依赖老员工口传。

## 场景故事

在中国移动 IDC 实习期间，我最深的感受不是设备多庞大，而是工单流转的摩擦力。一条"客户反馈机柜温度偏高"的工单，从受理台到分派给温控团队，中间要经过人工判断类型、查排班表找值班工程师、翻历史记录确认有没有同机柜的重复工单——夜班高峰期这一套走下来经常超过 20 分钟。更难受的是新人，不熟悉"过热先查冷通道封闭还是先查精密空调"这种隐性经验，只能反复问老员工。我想做的是一个能自己分类、自己查 SOP、自己判断要不要升级的工单 Agent——不是替代人，而是把"该找谁、该查什么、该不该立刻上报"这三个最耗时的决策自动化，让人去做只有人能做的事。

## 痛点拆解

| 痛点 | 表现 | 根因 |
|------|------|------|
| 分类靠经验 | 新人看到"温度偏高"不知道归温控还是电力 | 隐性知识未结构化 |
| 检索效率低 | 翻 200+ 份 SOP 手册找应急处置流程 | 知识检索未自动化 |
| 升级犹豫 | 不敢叫醒主管，也不敢不叫 | 缺少置信度驱动的升级规则 |
| 重复造轮子 | 同一机柜同类工单反复出现，每次都从头处理 | 历史工单未被利用 |

## 系统目标

一个基于 LangGraph 的 Multi-Agent 工单分派与诊断系统：

1. **自动分类**：输入工单文本，输出类别 + 优先级 + 置信度
2. **智能诊断**：调用监控/CMDB/历史工单三个工具辅助定位
3. **知识检索**：复用 P1 RAG 能力检索 SOP
4. **分级路由**：P0 直接升级、低置信度触发 Review 回环

## 不做什么

- 不替代人工执行（Agent 只输出方案，不直接操作设备）
- 不接入实时生产数据（监控/CMDB 使用 Mock 数据）
- 不做前端（终端 CLI + LangGraph visualizer 即可）
